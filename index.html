You Are Here Scope and Assumptions
1) Scope and Assumptions governance baseline
  • GitHub fully replaces LogicGate as the policy/standard system of record.
  • In-scope governance objects are Policy and Standard only.
  • Procedures are linked dependencies, outside Policy Team tollgate governance.
  • Tiered approvals require feature parity with PoP.
  • Approval matrix, extension limits, and checklist logic are config-driven and versioned.
2) Target Architecture platform topology
  • GitHub org dedicated to compliance governance.
  • Primary private repo: policy-sor.
  • Optional publication mirror: policy-public for Go/Policy delivery.
  • Automation stack: GitHub Actions + custom app policy-gatekeeper.

5) Access and Control Model security and merge controls

Required Branch protection + merge queue + signed commits + required checks.

  • Core teams: policy-authors, policy-team, compliance-leadership, legal-counsel.
  • CODEOWNERS enforces domain ownership by path.
  • policy-gatekeeper enforces dynamic tier/change approval rules.
6) Tollgate Workflow (TG1-TG4) end-to-end lifecycle
TG1 Intake
  • Triggered via intake issue form.
  • Runs field validation + duplicate/conflict precheck.
  • Scaffolds package + draft PR.
  • Exit: intake checks pass, domain/scope/template validated.
TG2 Revision
  • Runs template/style/schema checks and stakeholder evidence checks.
  • Generates redline + change summary artifacts.
  • Exit: required reviewers approve and checks pass.
TG3 QC
  • Policy Team checklist engine executes from governed config.
  • Produces QC report artifact and blocks critical failures.
  • Exit: policy-qc-pass + Policy Team approval.
TG4 Approval + Publication
  • Approval matrix resolves required approvers and evidence.
  • Merge blocked until approval-matrix-pass.
  • Post-merge: tag release, publish bundle, notify Slack, status published.
7) Approval Matrix Implementation tier parity engine

approval-matrix.yaml drives approvals by doc_type, tier, change_type, required approvers, external evidence needs, and escalation paths.

All merges are blocked unless matrix conditions pass through policy-gatekeeper.

8) Concurrent Reviewer Model parallel edits safely
  • One canonical PR per document lifecycle event.
  • Multiple reviewers can suggest edits simultaneously via review suggestions or child PRs.
  • Owner/Delegate acts as single integrator for accepted changes.
  • Conflicts are resolved in canonical branch before final approval.
  • Optional section-splitting reduces edit collisions on large documents.
9) Required GitHub Actions Checks merge gate checklist
  • policy-config-validation
  • policy-metadata-validation
  • policy-structure-style-check
  • policy-reference-link-check
  • policy-diff-summary
  • policy-qc-checklist
  • policy-approval-matrix
  • policy-publication-readiness

All checks are required on protected main.

10) Lifecycle Operations feature parity workflows
  • Scheduled reminders: 60/30/14/overdue.
  • Extension/deferment requests with approval and metadata update workflows.
  • Exception intake with controls, expiry, and renewal reminders.
  • Ownership transfer workflow with explicit acknowledgments.
  • Retirement workflow with approval gating and archiving signal.
11) Audit, Retention, and Evidence regulatory traceability
  • Evidence stored per version in approvals/ and evidence/.
  • Monthly governance report generated under reports/YYYY-MM/.
  • Immutable controls: protected branch, signed commits, no history rewrite.
  • Retention enforced from policy config and archival workflows.
12) Delivery Plan execution roadmap
  1. Phase 1 (4-6 weeks): repo scaffolding, metadata model, intake/revision workflows, base checks.
  2. Phase 2 (4-6 weeks): QC engine, approval matrix gatekeeper, publication pipeline.
  3. Phase 3 (3-5 weeks): exceptions/extensions/ownership/retirement + monthly reporting.
  4. Phase 4 (2-4 weeks): migration, parallel run, cutover, controls signoff.
