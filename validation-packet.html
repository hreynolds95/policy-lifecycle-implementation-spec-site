DMS Validation Packet

A reviewer-ready evidence checklist for proving the GitHub-native compliance DMS can replace LogicGate without losing PoP approval parity, audit traceability, or operational ownership.

{% include site-nav.html %}
Validation Outcome

Use this packet to decide if the system is cutover-ready.

Required Approval parity is proven.

Each PoP tollgate has a mapped GitHub control, owner, artifact, and failure mode.

Required Audit evidence is traceable.

Every published document can be traced to source, review, approval, merge, and publication evidence.

Required Operating ownership is assigned.

Daily, monthly, exception, orphan, migration, and stakeholder communication controls have DRIs.

Acceptance Checklist

Minimum evidence before LogicGate replacement

Validation Area Evidence To Review Pass Standard Primary Workstreams
System of record Repository taxonomy, registry schema, archive link model, and source-to-PDF trace. GitHub record is canonical for governed policies and standards. DMS-01, DMS-02, DMS-12
PoP tollgate parity Branch protection, CODEOWNERS, approval labels, QC checks, exception path, and merge blocks. Required approvers and checks cannot be bypassed for each tier. DMS-03, DMS-04, DMS-06
Audit chain Sample policy change PR, approval trail, generated output, audit log, and notification record. Reviewer can reconstruct who changed what, who approved it, when it published, and what was sent. Auditability, Policy Change Story, DMS-11
Portal usability Staff-facing library, search, document detail, role experience, and request initiation design. Employees can find current documents without GitHub training; governed actors know where to act. Role Experience, Two Surfaces, DMS-07
Continuous controls Review reminders, exception expiry, portfolio digest, orphan detection, and awareness communications. Recurring governance work produces structured records without manual chase-down. Automation Catalog, DMS-05, DMS-08, DMS-09
Cutover readiness Gap register, legacy migration proof, open decision log, roadmap, and owner-backed action plan. Known gaps are mitigated, accepted, or assigned before LogicGate decommission. Readiness Matrix, DMS-11, DMS-13, DMS-14
Reviewer Walkthrough

Sixty-minute validation agenda

00-10 min Frame the decision.

Use the executive summary to align on the LogicGate replacement assumption and decision ask.

10-25 min Trace one policy change.

Walk from intake through PR, QC, approval, publication, and evidence capture.

25-45 min Test the hard controls.

Review approval parity, template checks, owner restrictions, exception routing, and audit trail.

45-60 min Resolve cutover blockers.

Use the action plan to assign owners for unresolved Phase 2 inputs, dependencies, and gaps.

Decision Questions

Questions this packet should answer clearly

  1. Can a Tier 1, Tier 2, or Tier 3 policy change merge without the required approval evidence?
  2. Can the Policy Governance Team prove which document version was active on a historical date?
  3. Can employees find the current published document without needing GitHub training?
  4. Can exceptions, review reminders, orphaned documents, and monthly digests be monitored without spreadsheets?
  5. Are every Phase 2 blocker, decision, and legacy migration gap assigned to an owner?
