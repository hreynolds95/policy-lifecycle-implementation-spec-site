Automation Principle

Recurring governance work becomes scheduled evidence.

Each automation has a defined cadence, source signal, output record, and owner queue. The goal is not just notification. The goal is to create a durable record that shows the control ran, what it found, and what follow-up was required.

Cadences Daily / Weekly / Monthly

Controls run on a predictable schedule instead of relying on manual reminders.

Primary Output GitHub Issue

Each exception, reminder, orphan, or digest becomes a structured work item.

Evidence Audit Trail

Issues, labels, workflow logs, and linked records prove the control operated.

Catalog

Continuous automations in scope

Automation Cadence Trigger Signal Output Control Purpose
Review reminders Daily Document review date reaches 90, 60, or 30 days out. GitHub issue opened for upcoming review. Prevents stale policies and creates evidence that review owners were prompted.
Exception expiry alerts Daily Approved exception reaches 30 days before expiry. Renewal issue opened with exception context. Prevents expired exceptions from persisting without review.
Portfolio digest Monthly First day of each month. GitHub issue summarizing published docs, due reviews, overdue items, and open exceptions. Creates a recurring portfolio health snapshot for governance review.
Orphan detection Weekly Document has no active owner or owner signal is inactive. Issue opened for reassignment and escalation. Prevents documents from losing accountable ownership.
Record Design

Every automation should leave the same basic evidence shape.

01 Source signal

The registry, exception record, review date, or owner status that caused the automation to run.

02 Generated issue

The structured GitHub issue that captures owner, due date, context, and required follow-up.

03 Workflow log

The GitHub Actions run that proves when the control executed and whether it succeeded.

04 Closure evidence

The linked PR, decision, renewal, reassignment, or governance note that resolves the issue.
